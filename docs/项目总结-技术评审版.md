# AIroute · 项目总结（技术评审版）

> 面向：黑客松评委 / 技术评审。重点在创新点、架构、算法深度与可复现的量化证据。
>
> 本文中所有数字均来自项目实测产出（`data/eval/route_eval.md`、`data/eval/ranker_train.json`），可一键复跑。

---

## 0. 一句话定位

AIroute 是一个**基于约束优化 + 学习排序 + LLM Agent 三层架构的本地路线规划系统**：用户用自然语言说出诉求，系统在带时间窗的定向越野（OPTW）模型下产出**多条非支配（Pareto）路线**，每条路线都带可解释理由、UGC 证据、约束校验和**蒙特卡洛鲁棒性评估**，且全流程**有指标证明效果**。

---

## 1. 关键创新点

| # | 创新点 | 解决了什么问题 |
| --- | --- | --- |
| ① | **LLM 决策 + 工具白名单 + 确定性兜底**的 plan-act-observe Agent | LLM 只编排不直接造路线，杜绝幻觉路线；任何时候都能回退到可复现 pipeline |
| ② | **OPTW（带时间窗的定向越野）+ CP-SAT + 三档求解** | 把"路线规划"从启发式排序升级到带营业时间/预算/必到/类目组的硬约束最优化 |
| ③ | **加权标量化扫描的多目标 Pareto 前沿** | 同时给出"兴趣最高/省时/省钱/少排队/折中"5 条非支配解，可视化权衡而非强加 style |
| ④ | **训练/serve 同源的 Learning-to-Rank** | 用线上 `PoiScoringService` 直接产 breakdown 当训练特征，消除 train/serve skew |
| ⑤ | **蒙特卡洛路线鲁棒性评估** | 对随机排队/通行扰动跑 500 次模拟，输出"准时概率"作为路线质量第五维 |
| ⑥ | **跨会话长期记忆 + 情景召回** | `user_facts`（语义偏好）+ `session_vector_repo`（情景向量库）+ episodic summaries 三层记忆 |
| ⑦ | **全流程指标 + CI gate** | 5 个基准场景 + exact oracle 复解算 `route_quality_gap` + 解释忠实度自动化，可在 CI 上拦回归 |

### 2.1 关键 fallback 矩阵（"无 key 也能跑"是设计原则）

| 能力 | 主路径 | 降级路径 | 何时切换 |
| --- | --- | --- | --- |
| LLM 解析 | LongCat / OpenAI 兼容 | 关键字规则 | 无 key / 调用失败 / JSON 校验失败 |
| 路线优化 | OR-Tools CP-SAT | 精确枚举 → 贪心 | 候选 ≤8 走枚举；超时/无解走贪心 |
| Ranker | LightGBM LambdaMART | 规则加权总分 | 模型文件缺失或 `ranker_enabled=False` |
| 地图 | 高德实路网 | haversine + 速度估计 | 无 amap_key / 调用失败 |
| 检索 | FAISS + bge-small-zh | 关键字 + 池子 fallback | 无 index / `rag_enabled=False` |
| 鲁棒性 | Monte Carlo | 跳过（不强求） | sim 模块缺失或拒绝执行时 |

## 3. 核心技术深度

### 3.1 Agent Harness · plan-act-observe

`backend/app/agent/conductor.py`

- **强约束**：12 步预算、工具白名单（`schemas_for_llm()` 暴露的 11 个工具之外的请求直接拒绝）、每步结果作为 observation 写入 `AgentState.memory`，下一步基于状态推进。
- **LLM 决策与规则决策可切换**：`agent_tool_calling_enabled` 控制是否让 LLM 选下一步工具，`agent_fast_decision_enabled` 控制是否走规则快路径。两条路径在评测里都能跑出同样的约束满足率（gate PASS），证明规则路径足够安全可演示，LLM 路径用于深度交互。
- **Critic 多专家**：`Critic` 在终态前做幻觉检查（`hallucinated_poi/hallucinated_ugc`）和质量打分，可主动触发**一次**自动重试或路线压缩（`time_budget_exceeded` → `_compact_story_for_time_budget`）。这是"校验失败 → 自动补救 → 通过"的多步 trace 主路径。

### 3.2 OPTW · 带时间窗的定向越野

`backend/app/solver/optw.py`

- 形式化：在时间预算内从带 utility 的候选集中选子集并排序，**最大化总 utility**，硬约束 = 时间窗 + 营业时间 + 预算 + must-visit + 类目（含类目组 OR 关系）+ max_stops。
- CP-SAT 建模：流约束（开放路径，每点入度=出度=x_i）+ 时序传播（`OnlyEnforceIf` 保证 t_j ≥ t_i + visit_i + travel_ij）+ 营业时间窗（t_i ≥ open_i，t_i + visit_i ≤ close_i）。
- **多目标加权**：`weights={"utility": 1.0, "time": w_t, "cost": w_c, "queue": w_q}` 直接进入目标函数，Pareto 扫描时复用。
- **三档求解**：
  1. CP-SAT（默认，候选 ≤ 15）。
  2. 精确枚举（候选 ≤ 8 时也跑一遍作为 oracle，并产生 `optimality_gap`）。
  3. 贪心 fallback（超时 / 不可行时给"尽力而为"路线 + `constraint_violations` 标注）。

### 3.3 多目标 Pareto 前沿

`backend/app/solver/pareto.py`

- 5 个权重 profile：`interest / balanced / time_saving / budget_saving / low_queue`。
- 每个 profile 调一次 `solve_optw` 拿候选；候选数 ≤8 时还会跑全枚举补充候选池。
- 4 维支配判定（兴趣 max；时长/花费/排队 min）→ 去重 → 非支配过滤 → 按 `(-interest, cost, queue, time)` 排序 → 取前 5。
- 前端 `AmapRoutePage` 渲染卡片列表，tradeoff 文案由两解指标差**自动生成**（不再是硬编码 highlights）。

### 3.4 Learning-to-Rank · LightGBM LambdaMART

`backend/app/ml/ranker.py` + `scripts/train_ranker.py`

- 16 维特征：6 个规则打分项 + 5 个 penalty 项 + queue_min/price/distance_m/ugc_sim/popularity。
- **训练 / serve 一致性**：训练直接 `from app.services.poi_scoring_service import PoiScoringService` 调 `score_poi` 拿真实 breakdown 作为特征，消除原本"训练用占位 breakdown、线上用真实 breakdown"的 skew。
- **合成多 query 分组**：`SYNTH_QUERIES` 4 条不同诉求（少排队本地菜、情侣夜景、亲子室内、高效打卡），每条形成一个 ranking group，让 lambdarank 真正学到"相对排序"而不是退化成 pointwise。
- **NDCG 增益门控**：`should_enable_ranker` 要求 model NDCG@5 至少高过基线 3%；训练脚本输出 `ranker_enabled_recommended` 字段。
- **缺模型自动降级**：`PoiRanker.predict` 模型未加载时返回 `None`，`PoiScoringService._ranker_score` 把 None 当作"用规则总分"，无需重启切换。

### 3.5 蒙特卡洛鲁棒性

`backend/app/sim/montecarlo.py`

- 对最终 `RouteSkeleton` 做 N=500 次重采样，每站停留 / 排队 / 段间通行时间各加高斯扰动（cv≈0.2–0.25）。
- 输出 `on_time_prob` / `expected_overflow_min` / `p90_total_min`。
- 接入：`assess_robustness` 工具在 `validate_route` 之后、`critique` 之前自动跑一次，结果写入 `state.memory.robustness` 并挂到 `StoryPlan.robustness` 与 API 响应。

### 3.6 RAG 与多模态记忆

- **检索层**：FAISS + `BAAI/bge-small-zh-v1.5` 中文 embedding，带 `source_type` provenance（poi_profile / ugc_review / fts / feature_bucket），证据片段（`evidence_snippets`）回填到推荐理由。
- **三层记忆**（独立于会话存在）：
  1. **Episodic**：`session_summarizer` 摘要 + 上次会话 story 作为记忆 token 注入。
  2. **Semantic**：`user_facts`（喜欢的类目/区域/时间段、典型预算区间、被拒 POI）落 SQLite，按用户长期累积。
  3. **Vector**：`session_vector_repo` 把会话主诉求嵌入到 FAISS，下次进来 `recall_similar_sessions` 直接找语义相近的历史会话。
- Prometheus `MEMORY_LAYER_USAGE` 计数器分层统计三层的命中。

### 3.7 LLM 工程化

- `LlmClient.complete_tool_call` 走 OpenAI 兼容的 function calling，工具 schema 来自 `ToolRegistry.schemas_for_llm()`，非白名单工具被丢弃，回退到规则决策。
- LLM cache（`llm/cache.py`）针对幂等调用（intent / repair）缓存请求-响应对，降低成本。
- 全链路 OpenTelemetry span（每个工具一个 `tool.<name>` span，记录 session/trace/user/goal）+ structlog 结构化日志 + Prometheus 指标。
- `session_cost_summary(session_id)` 输出每会话的 token 总量、估算成本、各工具延迟排序，方便对比开 / 关 LLM tool-calling 的性能差异。

### 4.4 与所引论文的对应关系

`skills/local-route-agent/references/paper-insights.md` 列出的研究主线：

- **OPTW / PersTour（带时间窗的定向越野）** → `solver/optw.py` 的 CP-SAT 实现，objective 与约束精确对应。
- **GA / 迭代局部搜索（候选变大时的优化）** → 候选超 15 时 `_greedy_fallback` 做"破坏-修复"，未来可平滑替换成 ALNS。
- **I-AIR / ILSAP（多信号打分、Transformer-GCN）** → `ScoreBreakdown` 多信号契约 + `ml/ranker.py` LambdaMART 插入点，未来可换神经排序器。
- **Harness Agent 而非生成式 prompt** → `Conductor` 工具白名单 + 步数预算 + 规则兜底。
- **"Validate before explaining"** → `RouteValidator` + `Critic` 幻觉检查 + 评测中的解释忠实度自动化。

## 6. 演示亮点（3 个高光时刻）

**① "请给我安排一个少排队、能吃本地菜、顺路拍照的下午。"**
→ Agent 串行 9 个工具，10 秒内产出 5 条 Pareto 方案，每条带准时概率徽标。前端可点击切换观察 tradeoff 文案（"比当前多 8 兴趣、多 12 分时长、省 ¥40"）。

**② "改一下，我预算只剩 100，且不想去人最多的那个馆。"**
→ `/agent/adjust` 调起 `RepairAgent` 解析出 `category_hint + budget_per_person + rejected_poi`，重新求解 OPTW，新路线在 4 秒内返回，鲁棒性徽标随之更新。

**③ "为什么推荐这家咖啡店？"**
→ 路线每站的 `why_this_one` 都有 UGC 引证（`ugc_quote_ref` 指向 FAISS 检索到的具体评论）+ `score_breakdown` 各维度的数值。评测中"解释忠实度"自动化校验每条理由都对得上数据，分数 1.000。

## 7. 仍可扩展的方向

- **实时信号驱动**：天气 / 实时路况 / 实时排队 / 优惠券——本期主动跳过，是为了把验证体系先搭起来。接入后可重用现有 `event_type` 机制。
- **Pareto 切换重算路线**：当前 UI 切换是高亮 + 文案，下一步可一键 `createRouteChain(variant.ordered_ids)` 真切换。
- **多人组团偏好聚合**：把当前单 `preference_snapshot` 扩成多成员、引入 maxmin 公平或否决权，是个有意思的社会选择问题。
- **GA / ALNS 替换贪心 fallback**：候选超 15 时改用元启发式，与论文 ILS 对齐。
