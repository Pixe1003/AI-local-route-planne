# AIroute · 本地路线智能 Agent 系统

> 黑客松项目 / 主导设计与开发
> Python · React · OR-Tools · LightGBM · FAISS · LLM function calling

---

## 项目一句话

设计并实现了一个把自然语言诉求转化为「5 条 Pareto 路线 + 蒙特卡洛准时概率 + 可解释理由」的本地路线规划 Agent 系统。架构上把 **LLM 决策、约束优化、机器学习排序、离线评测** 拆成四层各司其职，每层都有指标守护和降级路径——本质区别于"LLM 直接吐 JSON"的 demo 类项目。

---

## 关键贡献

### 1. 设计并实现 Plan-Act-Observe Agent Harness

带 **12 步预算 + 11 工具白名单**的状态机驱动 agent（FastAPI + Pydantic + structlog）。LLM 通过 OpenAI 兼容 function calling 在工具间编排，工具执行结果作为 observation 写入 `AgentState.memory`；Critic 含**幻觉检查**（`hallucinated_poi` / `hallucinated_ugc`），失败可触发一次路线压缩或重组。LLM 决策默认走规则快路径，function calling 在岔路态可选启用——既保留 LLM 智能，又杜绝生成式幻觉。

### 2. 把路线规划形式化为 OPTW（带时间窗的定向越野），用 OR-Tools CP-SAT 求精确解

约束包含时间预算 + 营业时间窗 + 预算 + must-visit + 类目组（OR 关系），目标为最大化 Σutility。实现**三档求解**：CP-SAT MIP（默认）→ 精确枚举（候选 ≤ 8 作 oracle）→ 贪心 fallback。**5 个基准场景与精确解 gap = 0.001、约束违例为 0**。

### 3. 多目标 Pareto 前沿替代硬编码 style

5 个权重 profile（兴趣 / 折中 / 省时 / 省钱 / 少排队）通过 `ThreadPoolExecutor` 并行求解 → 4 维支配判定 → 非支配过滤。前端 React + TypeScript 渲染卡片切换，tradeoff 文案由两解指标差**自动生成**（不再硬编码）。**solve 时间通过并行降 28%**。

### 4. Learning-to-Rank 替换手调魔数评分

识别并修复了一个经典 ML 工程坑——**train/serve 特征偏差**（训练用占位 breakdown vs 线上用真实 breakdown）：改为训练直接调用线上 `PoiScoringService.score_poi` 产真实特征，同时合成多 query 分组让 LightGBM LambdaMART 真做 listwise 排序。**NDCG@5 从 0.2422 提升至 0.4725（+95.1%）**，并设置基于增益的自动启用门控；缺模型时自动降级到规则总分，零事故升级。

### 5. 离线评测 Harness + CI Gate

5 个场景 YAML × 6 项指标：约束满足率 / **解释忠实度** / 与 CP-SAT 最优解 gap / NDCG / Pareto 变体数 / 蒙特卡洛准时概率。其中**解释忠实度自动化**——解析每条推荐理由提取 claim → 回查 `score_breakdown` / POI 属性 / UGC 关键词 → 输出一致性得分。接 GitHub Actions PR gate，gate 红即 PR 红。**把"LLM 应用"从 demo 升级到"指标可证、回归可拦"**。

### 6. 蒙特卡洛鲁棒性

对最终路线做 500 次随机采样（每站停留 / 排队 / 段间通行时间高斯扰动），输出 `on_time_prob / expected_overflow_min / p90_total_min`。Conductor 在 validate 后自动跑，前端徽标显示"准时 95% · P90 4h20"。**平均准时概率 0.951**。

### 7. 三层记忆 + 跨会话情景召回

`agent_sessions` SQLite 按 user_id 隔离；`user_facts` 持久化长期偏好（喜欢的类目 / 区域 / 预算区间 / 被拒 POI）；`session_vector_repo` 把会话主诉求向量化到 FAISS，下次进来 `recall_similar_sessions` 找语义相近历史。Prometheus 指标分层统计三层命中率。

### 8. 端到端可观测

structlog 结构化日志 + Prometheus 11 项指标（含 `tool_latency` / `agent_run_latency` / 缓存命中 / 幻觉计数 / 三层记忆使用）+ OpenTelemetry span（每工具一个）+ `session_cost_summary` 按会话核 token / 延迟 / 估算 USD。

### 9. 启动 warmup probe

FastAPI `lifespan` 启动钩子预加载 FAISS / bge-small-zh / LightGBM / SQLite，**把 ~30 s 模型加载从首个真实用户挪到部署阶段，cold start 从 126s → 4s**。

### 10. 诚实的工程纪律

尝试过 CP-SAT 主求解 timeout 3 → 1.5s 优化（OPT-5），实测对 p95 无改善且引入 eval gap 微退，**主动撤回并在文档完整记录"试过、无效、撤回"全过程**。延迟阈值 advisory 从 3000 → 4000 → 4500 ms 演化，每一步都有数据依据。这是和"所有 KPI 强行染绿"的项目最大区别。

## 思路落地应用的价值

这套架构不只是黑客松 demo，而是回答了一个在大厂 LLM 应用落地中反复出现的根本问题：**怎么让 LLM 系统的输出"对得起约束、说得清依据、量得到效果"？**

- **对得起约束**——LLM 不负责直接生成路线，只负责编排；真正的路线来自 OPTW 模型在硬约束下的最优解。这把"约束满足"从"祈祷 LLM 听话"变成"求解器返回的合法解"，**约束违例从 demo 时的常见状态变成结构性消除**。
- **说得清依据**——每条推荐理由都对应 `score_breakdown` 数值 + UGC 引证片段 + provenance 标签。解释忠实度**自动化校验**确保理由不是 LLM 编的。这一项是大多数项目漏掉、但产品落地后用户最先质疑的。
- **量得到效果**——完整的离线评测 + CI gate 让"路线变好了"成为**可证明的数字**，而不是"我感觉变好了"。NDCG +95%、约束满足率 100%、与最优解 gap 0.001——任何后续优化都能在同一基准上比 before/after。

### 可迁移性

这套思路可以**直接迁移到任何高约束 + 高可解释要求的 LLM 应用场景**：

- **智能客服决策** —— 约束 = 工单流程 / SLA / 合规；可解释 = 合规依据回溯
- **医疗辅助诊断** —— 约束 = 用药禁忌 / 检查互斥；可解释 = 证据链回溯
- **企业知识检索** —— 约束 = 权限边界 / 信息分类；可解释 = 源文档引用
- **金融风控决策** —— 约束 = 法规要求 / 风险阈值；可解释 = 因子贡献可审计

**LLM 负责"决策与组织"，专用引擎（运筹 / ML / 规则）负责"硬约束求解"，评测体系负责"效果守门"** —— 这是把 LLM 从"会聊天的玩具"做成"能上线的系统"的工程方法论。
