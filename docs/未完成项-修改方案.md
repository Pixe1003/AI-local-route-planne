## WP-3 · LTR：从「管线就绪」到「训练可信 + 默认启用 + 进评测」

### 问题（读 `scripts/train_ranker.py` + `poi_scoring_service.py` 后定位）

1. **train/serve 特征偏差**：训练时 `build_features(poi, _rule_breakdown(poi), distance_m=0, ugc_sim=0)`，而线上 `_ranker_score` 用的是真实 `score_poi` 的完整 `breakdown` + 真实 `distance` + 由 `ugc_match` 推导的 `ugc_sim`。两边的 16 维分布不一致，模型学到的权重在线上失真。
2. **弱标签太弱**：`_weak_label` 只看 rating/review/category/queue，完全没用到用户偏好信号（`preference_snapshot` 的 liked/disliked）。
3. **单 group 退化**：`group=[len(train_rows)]` 把所有 POI 当成一个 query 的候选，lambdarank 退化；`baseline_ndcg` 用 `row[0]+row[1]` 当基线分是 hack。
4. **未进评测**：`ndcg_at_k` 只在训练脚本里算一次，没进 `backend/eval`，无法持续对比 before/after。

### 修改清单

- 改：`scripts/train_ranker.py`（特征对齐 + 多 query 分组 + 偏好标签 + 真实基线）。
- 改：`backend/app/services/poi_scoring_service.py`（确保 `_ranker_score` 的 `ugc_sim/distance_m` 与训练同源；抽一个 `feature_context` helper 两边共用）。
- 改：`backend/eval/metrics.py` + `run_eval.py`（NDCG@5 进聚合，见 WP-6）。
- 改：`backend/app/config.py` / 部署：训练达标后置 `ranker_enabled=True`。
- 测试：`backend/tests/test_ranker.py` 增「训练特征维度 == 线上特征维度」「缺模型降级」用例。

### 关键改动：消除 train/serve 偏差

在 `ml/features.py` 增一个统一的「特征上下文」构造，训练与线上都走它：

```python
# ml/features.py  追加
from dataclasses import dataclass

@dataclass
class FeatureContext:
    distance_m: float = 0.0
    ugc_sim: float = 0.0

def ugc_sim_from_match(ugc_match: float) -> float:
    # 与 poi_scoring_service._ranker_score 当前的推导保持一致，集中到一处
    return max(0.0, min((ugc_match - 6.0) / 10.0, 1.0))
```

`poi_scoring_service._ranker_score` 改为复用它（去掉散落的内联计算）：

```python
# poi_scoring_service.py
from app.ml.features import build_features, ugc_sim_from_match

def _ranker_score(self, poi, breakdown, context, ugc_match):
    settings = get_settings()
    if not settings.ranker_enabled:
        return None
    distance = self._raw_distance_m(poi, context) or 0.0
    features = build_features(poi, breakdown,
                              distance_m=distance,
                              ugc_sim=ugc_sim_from_match(ugc_match))
    return get_ranker(settings.ranker_model_path).predict(features)
```

```python
# 形成真正的 ranking group；标签融合规则分桶 + 偏好信号。

SYNTH_QUERIES = [
    {"text": "少排队 本地菜 拍照", "liked_categories": {"restaurant", "scenic"}},
    {"text": "情侣 夜景 咖啡 轻松", "liked_categories": {"cafe", "nightlife"}},
    {"text": "亲子 室内 文化 不赶", "liked_categories": {"culture", "shopping"}},
    {"text": "高效 多逛 地标", "liked_categories": {"scenic", "culture"}},
]

def _build_dataset(pois):
    rows, labels, groups = [], [], []
    scorer = PoiScoringService()                      # 直接用线上评分器产出 breakdown，彻底消除偏差
    for q in SYNTH_QUERIES:
        group_rows = 0
        for poi in pois:
            breakdown = scorer.score_poi(poi, free_text=q["text"])   # 真实 breakdown
            rows.append(build_features(poi, breakdown,
                                       distance_m=0.0,
                                       ugc_sim=ugc_sim_from_match(breakdown.ugc_match)))
            labels.append(_query_label(poi, q))       # 0~3 相关性
            group_rows += 1
        groups.append(group_rows)
    return rows, labels, groups

def _query_label(poi, q) -> int:
    label = 0
    if poi.category in q["liked_categories"]:
        label += 2
    if getattr(poi, "rating", 0) >= 4.5:
        label += 1
    if (getattr(poi, "queue_estimate", {}) or {}).get("weekend_peak", 0) <= 25:
        label += 1
    return min(label, 4)
```

## WP-5 · 多目标 Pareto 前沿（全新）

### 设计

当前 agent 只产出**一条** story 路线（`solve_constrained_route → compose_story`）。Pareto 方案 = 围绕现有 `solve_optw` 做**加权标量化扫描**：对「兴趣 / 时长 / 花费 / 排队」四个目标取多组权重各解一次，得到若干路线，做**非支配过滤**后挑代表解返回。

这需要两步底层改动：给 `OptwNode` 补 `queue_min`、让 `solve_optw` 目标支持权重。

### 改动清单

- 改：`backend/app/solver/optw.py`（`OptwNode` 加 `queue_min`；目标加权惩罚 time/cost/queue）。
- 新增：`backend/app/solver/pareto.py`（权重网格 → 多次求解 → 非支配过滤 → 代表解）。
- 改：`backend/app/agent/tools.py`（`_solve_constrained_route` 末尾产出 `route_variants` 存入 memory；`OptwNode` 构造补 `queue_min`）。
- 改：`backend/app/agent/state.py`（`AgentMemory` 加 `route_variants: list[dict]`）。
- 改：`backend/app/api/routes_agent.py`（`AgentRunResponse` 加 `route_variants`）。
- 改：前端 `frontend/src/components/PlanCompare.tsx`（前沿散点 + 方案切换；tradeoff 文案由指标差生成）。
- 测试：`backend/tests/test_pareto.py`（代表解互不支配、覆盖度）。

### `solve_optw` 多目标化（最小侵入）

```python
# solver/optw.py
@dataclass(frozen=True)
class OptwNode:
    poi_id: str
    category: str
    utility: float
    visit_min: int
    price: int
    open_min: int
    close_min: int
    queue_min: int = 0            # 新增

# solve_optw(...) 新增参数：weights: dict | None = None
# 现有目标：model.Maximize(sum(int(round(node.utility*100)) * x[i] ...))
# 改为按权重做标量化（utility 为正向，其余为惩罚）：
w = {"utility": 1.0, "time": 0.0, "cost": 0.0, "queue": 0.0} | (weights or {})
util_term  = sum(int(round(node_index[i].utility * 100)) * x[i] for i in range(1, count + 1))
cost_term  = sum(node_index[i].price * x[i] for i in range(1, count + 1))
queue_term = sum(node_index[i].queue_min * x[i] for i in range(1, count + 1))
# t_end 已是 makespan（到达+停留的最末时刻）
model.Maximize(
    int(w["utility"] * 100) * util_term
    - int(w["time"]  * 100) * t_end
    - int(w["cost"]  * 100) * cost_term
    - int(w["queue"] * 100) * queue_term
)
```

> 贪心兜底 `_greedy_fallback` 也按同一权重排序（把排序 key 从 `utility` 换成 `utility - w_time*… - w_cost*…`），保证降级时风格一致。

### `solver/pareto.py`（全新）

```python
# backend/app/solver/pareto.py
from dataclasses import dataclass
from app.solver.optw import OptwNode, OptwResult, solve_optw

WEIGHT_GRID = [
    {"utility": 1.0},                                   # 纯兴趣
    {"utility": 0.7, "time": 0.3},                      # 偏省时
    {"utility": 0.7, "cost": 0.3},                      # 偏省钱
    {"utility": 0.7, "queue": 0.3},                     # 偏少排队
    {"utility": 0.5, "time": 0.2, "cost": 0.2, "queue": 0.1},  # 折中
]

@dataclass
class RouteVariant:
    label: str
    ordered_ids: list[str]
    interest: float
    time_min: int
    cost: int
    queue_min: int
    solver: str

def _metrics(nodes_by_id, result: OptwResult) -> dict:
    sel = [nodes_by_id[i] for i in result.ordered_ids]
    return {"interest": result.selected_utility,
            "time": result.total_duration_min,
            "cost": result.total_cost,
            "queue": sum(n.queue_min for n in sel)}

def _dominates(a, b) -> bool:
    ge = (a["interest"] >= b["interest"] and a["time"] <= b["time"]
          and a["cost"] <= b["cost"] and a["queue"] <= b["queue"])
    gt = (a["interest"] > b["interest"] or a["time"] < b["time"]
          or a["cost"] < b["cost"] or a["queue"] < b["queue"])
    return ge and gt

def build_pareto_variants(nodes, travel, *, solve_kwargs) -> list[RouteVariant]:
    nodes_by_id = {n.poi_id: n for n in nodes}
    seen, raw = set(), []
    for weights in WEIGHT_GRID:
        res = solve_optw(nodes, travel, weights=weights, **solve_kwargs)
        key = tuple(res.ordered_ids)
        if not key or key in seen:
            continue
        seen.add(key)
        raw.append((weights, res, _metrics(nodes_by_id, res)))
    # 非支配过滤
    front = [item for item in raw
             if not any(_dominates(o[2], item[2]) for o in raw if o is not item)]
    return [_to_variant(w, r, m) for (w, r, m) in front][:5]
```

`tools.py:_solve_constrained_route` 末尾调用 `build_pareto_variants`，把结果 `model_dump` 后写入 `memory_patch={"route_variants": [...]}`，并把第一条仍作为主路线（保持现有行为不破坏）。

### 验收

- `test_pareto.py`：返回的 variants 两两不支配（自动断言）；至少能产出「偏省钱 / 偏省时 / 折中」可区分的 3 条。
- 前端 tradeoff 文案来自两解指标差（如「比折中方案省 ¥40、多花 18 分钟」），不再写死。

### route_quality_gap 的 oracle 复核

思路：用场景产出的候选池重建 `OptwNode`，以 `solver_mode="exact"` 再解一次取**最优 utility**，与 agent 实际选中的 utility 比。

```python
# backend/eval/run_eval.py  —— 在 _evaluate_scenario 内补充
from app.solver.optw import solve_optw

opt = (story or {}).get("optimization") or {}
selected_utility = float(opt.get("selected_utility") or 0.0)

# 复用 agent 返回的 pool 重建节点（与 tools._solve_constrained_route 同口径）
nodes, travel, solve_kwargs = _rebuild_optw_inputs(payload, scenario)
oracle = solve_optw(nodes, travel, solver_mode="exact", **solve_kwargs)
gap = None
if oracle.selected_utility > 0:
    gap = round(max(0.0, (oracle.selected_utility - selected_utility) / oracle.selected_utility), 4)

return EvalResult(..., route_quality_gap=gap)
```

> `_rebuild_optw_inputs` 把 `payload["pool"]` 的 POI 映射成 `OptwNode`（含本次新增的 `queue_min`），`solve_kwargs` 用场景的时间窗/预算/必含类目。与生产同一函数口径，避免口径漂移。

### `sim/montecarlo.py`（全新）

```python
# backend/app/sim/montecarlo.py
import random
from dataclasses import dataclass

@dataclass
class RobustnessSummary:
    on_time_prob: float
    expected_overflow_min: float
    p90_total_min: int
    samples: int

def simulate(skeleton, queue_by_id: dict[str, int], end_min: int,
             *, n: int = 500, seed: int = 42, cv: float = 0.3) -> RobustnessSummary:
    rng = random.Random(seed)
    start_min = _to_min(skeleton.stops[0].arrival_time) if skeleton.stops else 0
    overflows, on_time = [], 0
    for _ in range(n):
        t = start_min
        for stop in skeleton.stops:
            base_q = queue_by_id.get(stop.poi_id, 0)
            t += stop.duration_min + max(0, rng.gauss(base_q, base_q * cv))
            if stop.transport_to_next:
                leg = stop.transport_to_next.duration_min
                t += max(1, rng.gauss(leg, leg * cv))
        overflow = max(0, t - end_min)
        overflows.append(overflow)
        on_time += (overflow == 0)
    overflows.sort()
    return RobustnessSummary(
        on_time_prob=round(on_time / n, 3),
        expected_overflow_min=round(sum(overflows) / n, 1),
        p90_total_min=int(start_min + overflows[int(n * 0.9) - 1]),
        samples=n,
    )

def _to_min(hhmm: str) -> int:
    h, m = hhmm.split(":")
    return int(h) * 60 + int(m)
```
