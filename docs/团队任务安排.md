# AI 本地路线智能规划系统团队任务安排

## 1. 项目目标

本项目面向美团开发比赛题目「AI 本地路线智能规划」，目标是在比赛周期内完成一个可演示的本地智能路线规划系统。系统需要支持用户输入出行需求，自动生成 POI 推荐池，用户勾选后生成多条差异化路线，并支持对话式调整。

由于目标用户大概率在出行前或出行中使用手机完成决策，产品体验以移动端优先设计，桌面端作为答辩演示和开发调试补充。

项目交付重点不是做一个“大而全”的旅游 App，而是清楚展示三件事：

- AI 能理解用户的出行偏好和约束；
- 系统能把 POI、UGC、时间、空间距离转化为可执行路线；
- Demo 稳定、流程完整、答辩叙事清楚。

## 2. 团队角色分工

### A 同学：整体框架与项目统筹负责人

A 同学负责把控项目整体方向、模块边界、系统架构和最终交付质量，确保每个人的工作能拼成一个完整 Demo。

主要职责：

- 维护整体技术架构和模块边界；
- 统筹前后端接口契约、数据 schema、开发节奏；
- 负责 LLM/Agent 整体链路设计，包括意图理解、方案生成、对话调整的框架；
- 把控产品逻辑是否完整：输入、推荐、选择、规划、调整、解释；
- 组织联调、测试、Demo 演练和答辩材料整合；
- 在功能取舍上做最终判断，避免比赛后期范围失控。

重点交付：

- 项目整体架构说明；
- API/schema 对齐文档；
- Agent 工作流与核心 prompt 设计；
- Demo 主流程检查清单；
- 答辩 PPT 主线和最终材料整合。

对应模块：

- `backend/app/services/intent_service.py`
- `backend/app/services/plan_service.py`
- `backend/app/services/chat_service.py`
- `backend/app/llm/prompts/`
- `README.md`
- `docs/`

### B 同学：城市规划逻辑与路线合理性负责人

B 同学负责把城乡规划背景转化为项目的差异化能力，保证路线不是随机串点，而是符合真实城市出行逻辑。

主要职责：

- 设计 POI 分类体系和路线评价维度；
- 定义不同路线风格的规划逻辑，例如高效打卡、松弛漫游、美食优先；
- 负责时间窗、空间距离、通勤方式、停留时长等约束逻辑；
- 检查路线是否符合真实城市动线；
- 准备和校验上海 POI seed 数据、UGC mock 数据；
- 为答辩提供“为什么这条路线合理”的专业解释。

重点交付：

- POI 分类和标签体系；
- 路线约束与评价规则；
- 三类路线风格的差异化策略；
- 高质量 POI/UGC 示例数据；
- 路线合理性验证案例。

对应模块：

- `backend/app/repositories/seed_data.py`
- `backend/app/solver/styles.py`
- `backend/app/solver/distance.py`
- `backend/app/services/solver_service.py`
- `data/processed/`

### C 同学：推荐池与用户入口负责人

C 同学负责用户进入系统后的第一段核心体验：用户如何表达需求，以及系统如何生成一个有吸引力、可选择的 POI 推荐池。

主要职责：

- 设计用户入口页面，包括城市、时间、persona、预算、自由文本输入；
- 负责用户入口和推荐池页面的移动端优先适配；
- 负责推荐池生成逻辑：召回、过滤、排序、分类、默认勾选；
- 负责推荐理由、UGC 金句、关键词、排队预估等信息在前端的展示；
- 优化用户从“输入需求”到“选择 POI”的体验；
- 确保推荐池能稳定返回 15-20 个候选，并默认勾选 3-5 个；
- 和 A 同学对齐推荐池 schema，和 B 同学对齐 POI 分类及空间合理性。

重点交付：

- 用户入口页；
- 推荐池 API 和前端展示；
- POI 卡片组件；
- 移动端推荐池浏览和底部主操作体验；
- 推荐池默认勾选策略；
- 推荐理由和 UGC 证据展示。

对应模块：

- `frontend/src/pages/HomePage.tsx`
- `frontend/src/pages/PoolPage.tsx`
- `frontend/src/components/PoiCard.tsx`
- `frontend/src/components/PoolGrid.tsx`
- `frontend/src/store/poolStore.ts`
- `backend/app/services/pool_service.py`
- `backend/app/api/routes_pool.py`

## 3. 模块协作边界

### 推荐池链路

负责人：C 同学  
协作：A 同学负责 schema 和 Agent 逻辑，B 同学提供 POI 标签与路线视角建议。

输入：

- 用户城市、日期、时间窗；
- persona 标签；
- 预算；
- 自由文本偏好。

输出：

- `PoolResponse`
- 分类 POI 列表；
- 默认勾选 POI；
- 推荐理由；
- UGC 金句；
- 排队时间预估。

### 路线规划链路

负责人：B 同学  
协作：A 同学负责整体编排，C 同学保证前端能消费结果。

输入：

- 用户勾选的 POI；
- 时间窗；
- 预算；
- 用户偏好；
- POI 距离和停留时长。

输出：

- 三条差异化路线；
- 每站到达/离开时间；
- 通勤时间和距离；
- 被舍弃 POI 及原因；
- 方案指标，例如总时长、总花费、站点数、排队时间。

### Agent 与对话调整链路

负责人：A 同学  
协作：B 同学提供调整后的路线合理性判断，C 同学提供前端交互入口。

支持的调整意图：

- 替换某站，例如“把第二站换成不排队的”；
- 增加一站，例如“加一杯咖啡”；
- 删除一站，例如“跳过下一站”；
- 压缩时间，例如“剩下能不能快点”。

输出：

- 调整意图类型；
- 更新后的路线；
- 给用户的自然语言回复；
- 是否需要用户确认。

## 4. 开发节奏建议

### 第 1 阶段：主链路稳定

目标：系统能从首页输入走到推荐池，再生成路线。

- A 同学：确认 schema、API、整体架构和主流程；
- B 同学：整理 POI 数据、分类、停留时长、路线风格规则；
- C 同学：完成用户入口和推荐池页面。

验收标准：

- 首页可以提交用户需求；
- 推荐池能返回 15 个以上 POI；
- 默认勾选 3-5 个；
- 能生成至少 1 条可展示路线。

### 第 2 阶段：差异化和解释性

目标：让产品看起来“真的有 AI 决策能力”。

- A 同学：优化意图解析、方案润色、对话调整；
- B 同学：让三条路线在站点、顺序、节奏上有明显差异；
- C 同学：优化 POI 卡片、推荐理由、UGC 金句展示。

验收标准：

- 能生成 3 条路线；
- 三条路线有明显差异；
- 每个方案有 tradeoffs；
- 每个站点有 UGC 证据或推荐理由。

### 第 3 阶段：Demo 打磨

目标：稳定演示，准备答辩。

- A 同学：组织端到端测试、整理 PPT 主线；
- B 同学：检查路线合理性，并准备规划专业解释；
- C 同学：修前端体验问题、接口联调问题和移动端适配问题。

验收标准：

- 标准 Demo case 可以连续跑通；
- LLM 或地图失败时有兜底；
- 页面无明显空白、错位、报错；
- 手机视口下首页、推荐池页、路线页都能自然完成主流程；
- 三个人都能独立讲清楚自己负责的部分。

## 5. 每日协作机制

建议每天进行一次 15 分钟同步：

- 昨天完成了什么；
- 今天要完成什么；
- 当前是否有阻塞；
- 是否影响其他人的接口或数据结构。

接口变更规则：

- 任何字段名、字段类型、API 返回结构变更，必须先同步；
- 不允许单人在本地随意改 schema；
- 如果必须改，A 同学负责更新文档并通知 B/C。

联调规则：

- 每个模块完成后必须提供一个可复现输入；
- 每个 API 至少保留一个标准测试 case；
- 比赛前一天不新增功能，只修 bug 和准备 Demo。

## 6. 答辩分工

### A 同学

讲清楚：

- 项目整体问题定义；
- AI/Agent 如何理解用户需求；
- 系统架构和主链路；
- Demo 流程和产品价值。

### B 同学

讲清楚：

- 城市规划视角如何进入路线生成；
- POI 分类、时间窗、空间距离、停留时长如何影响路线；
- 为什么三条路线各自合理；
- 项目相比普通搜索推荐的差异化。

### C 同学

讲清楚：

- 用户入口如何降低表达成本；
- 推荐池如何生成和排序；
- 前端如何展示 POI 证据、推荐理由和默认勾选；
- 系统如何保证 Demo 可用和稳定。

## 7. 最终交付清单

- 可运行代码仓库；
- 前后端主链路；
- 推荐池与路线生成 API；
- 用户入口、推荐池页、路线展示页；
- 移动端适配后的核心页面；
- 至少 50 个 POI seed 数据；
- 至少 3 条可演示路线；
- 对话调整能力；
- README 和启动命令；
- Demo 视频或备用截图；
- 答辩 PPT。

## 8. 一句话分工总结

- A 同学负责“整体框架、Agent 链路和最终交付”；
- B 同学负责“城市空间逻辑、POI 数据和路线合理性”；
- C 同学负责“用户入口、推荐池和选择体验”。
