# AIroute · 响应速度测试设计

> 这份文档说明 `scripts/bench_latency.py` 的设计意图、变量矩阵、指标定义和已知偏差，便于后续读报告和迭代基准。

---

## 1. 测什么

主问题：**用户发出一个自然语言诉求到拿到带 Pareto 候选 + 鲁棒性评估的路线，需要多少时间？**

拆成 4 个可量化的子问题：

| # | 子问题 | 报告里的位置 |
| --- | --- | --- |
| Q1 | 端到端 `/api/agent/run` 的稳态延迟（用户每次发请求要等多久） | §1 E2E summary |
| Q2 | 第一次启动 vs 缓存预热后差多少（一次性模型加载 / FAISS 预热的代价） | §2 Cold start |
| Q3 | LLM 自主决策开 vs 关，多出来的成本是什么量级 | §1 §3 跨模式比较 |
| Q4 | 哪个工具在拖慢主链路（找瓶颈） | §4 Per-tool latency |

---

## 2. 变量矩阵

```
┌────────────────────────────────────────────────────────────┐
│  Scenarios × Modes × Repeats × {cold | warm}               │
│                                                            │
│  Scenarios = 5  (backend/eval/scenarios/*.yaml)            │
│  Modes     = {rule, llm}                                   │
│  Repeats   = 5  (默认；首轮 = cold，余 4 轮 = warm)        │
│                                                            │
│  默认总计: 5 × 2 × 5 = 50 次 /agent/run                    │
└────────────────────────────────────────────────────────────┘
```

- **Scenarios**：直接复用评测的 5 个基准场景（半日吃饭 / 预算紧 / 必去景点 / 少排队 / 家庭雨天），避免出现"延迟用一套场景、效果用另一套"的口径漂移。
- **Modes**：通过 `AGENT_FAST_DECISION_ENABLED` 环境变量翻位，每次翻位后 `get_settings.cache_clear()` 立即生效。
  - `rule`：Conductor 走规则快路径，不调 LLM。
  - `llm`：Conductor 走 function calling 让 LLM 选下一步工具。
- **Cold vs warm**：每个 (mode, scenario) cell 内第一次调用单独记为 cold，后续 N-1 次平均为 warm。

## 3. 指标定义

| 指标 | 怎么算 | 为什么 |
| --- | --- | --- |
| **E2E latency** | `time.perf_counter()` 包住 `client.post(...)` 全程 | 最接近用户感知 |
| **Cold start** | 该 cell 第一次调用的 E2E 延迟 | 包含一次性的模型加载 / 缓存填充 |
| **Warm p50 / p95 / p99** | N-1 个 warm 样本的分位数 | 看稳态、长尾 |
| **Per-tool mean** | `response.steps[*].latency_ms` 按 `tool_name` 聚合，**只统计 warm 轮** | 找瓶颈工具 |

> p99 在默认 `repeats=5` 下样本太少，仅做参考。需要严肃 tail-latency 数据时把 `--repeats` 调到 ≥ 20。

报告还会输出一组**建议达标线**，只用于读数和后续优化排序，不作为脚本退出码：

| 建议线 | 口径 | 说明 |
| --- | --- | --- |
| **Warm p95 ≤ 4500ms** | 各 mode 下所有场景 warm 样本合并后计算 | 判断后端主链路稳态是否足够快。阈值演变记录：(1) 早期 3000ms 基于 `repeats=2` 的单样本草稿（warm p95 = 2598ms）；(2) `repeats=20` 首跑得到 3555ms，曾短暂收紧到 4000ms；(3) OPT-5 试验期间和回退后两次独立跑都稳定在 4290ms，证明 3555 是偏乐观的单点观测，真实稳态在 ~4.3s 区间；(4) 最终阈值 = 4290 + ~5% 噪声头 ≈ 4500ms。要再压需要算法级改动（减少 Pareto profile / 拆 recommend_pool / CP-SAT 多核搜索） |
| **Cold ≤ Warm p95 × 3** | 各 mode 最大 cold 与该 mode warm p95 对比 | 判断首次调用是否有明显预热问题 |

即使超过建议线，`bench_latency.py` 也只在报告里标 `WARN`，不会让命令失败；如果需要 CI 硬门禁，应另开设计讨论阈值、运行环境和误报成本。

## 4. 隔离变量的工程手段

让对比结果可信，要把"非待测变量"压住：

1. **环境隔离** — 通过 `LOCAL_ROUTE_DISABLE_ENV_FILE=1` 跳过本地 `.env`，避免本机 LLM key / Amap key 让两次运行结果不一致。
2. **网络隔离** — 用 `_patch_local_amap_client()` 把高德 client 换成 in-process 桩，和 `backend/eval/run_eval.py` 同一套补丁，保证测出来的是 **Agent + Solver 主链路**，不是高德 API 抖动。
3. **进程内调用** — 用 FastAPI `TestClient`（不是 `httpx + uvicorn`），少一层 socket / asyncio loop 切换的噪音，让样本之间方差更小。
4. **缓存清理** — 每次切 mode 后 `get_settings.cache_clear()` + `get_ranker.cache_clear()` + `get_poi_repository.cache_clear()`，把可清的 lru_cache 都清掉再开始计时。

---

## 5. 不能完美隔离的部分（已知偏差）

诚实记录以免误读：

- **FAISS / sentence-transformer 模型句柄**进程级常驻，不能 unload。所以"cold for the agent state machine" ≠ "cold for the OS"。要测 OS 级冷启动，得用 subprocess 拉起新进程。
- **首个 mode 享受真正的进程冷启动，第二个 mode 享受半热缓存**。两次跑同一脚本应该按相同 mode 顺序，避免对比偏移。
- **没有 LLM key 时，`llm` 模式实际上还是走规则路径**（LlmClient 内部 fallback），所以 llm 与 rule 的延迟会接近——这不是 bug，是 fallback 设计本身。报告会显式标注 `LLM_API_KEY` 是否配置。
- **首次跑 `repeats=5` 时 p99 信息量很弱**，因为只有 4 个 warm 样本。脚本里照样输出，但用作"对比"而非"绝对承诺"。

## 6. 如何运行

最常用：

```powershell
# 默认 5 场景 × 2 模式 × 5 次（首次冷启），默认输出 data/eval/latency_report.md
python scripts\bench_latency.py
```

冒烟版（更快，看脚本是否正常）：

```powershell
python scripts\bench_latency.py --repeats 2
```

只跑一种 mode（典型 CI 场景）：

```powershell
python scripts\bench_latency.py --modes rule --repeats 3
```

严肃 tail-latency 测量：

```powershell
python scripts\bench_latency.py --repeats 30
```

带真实 LLM key（看 LLM 决策真实开销）：

```powershell
# 把 .env 重新打开（或直接 export 关键变量）
$env:LOCAL_ROUTE_DISABLE_ENV_FILE = "0"
python scripts\bench_latency.py --modes llm --repeats 10
```

## 7. 报告的解读路径（建议看的顺序）

打开 `data/eval/latency_report.md` 后按 §1 → §5 → §2 → §4 → §3 看：

1. **§1 E2E summary** ——一眼看到稳态 p50/p95；这是回答"产品现在快不快"的核心数字。
2. **§5 Advisory thresholds** ——看 warm p95 和 cold ratio 是否触发 `WARN`；这是建议线，不是 CI 门禁。
3. **§2 Cold start** ——看首次调用的代价。如果你担心冷启动慢，这一段就是答案；正常应在 p95 的 2–3 倍以内。
4. **§4 Per-tool latency** ——按工具找瓶颈。一般 `solve_constrained_route`（CP-SAT，含 Pareto 多次求解）、`search_ugc_evidence`（FAISS 检索）、`get_amap_chain` 排前三。
5. **§3 Per-scenario E2E** ——查"某一场景特别慢"的反常点，多半是 must-visit / 类目组导致 CP-SAT 收敛慢。

## 8. 改进路径（如果某个数字不达标）

把这个表挂在 bench 报告下方，作为后续工作的清单：

| 症状 | 可能根因 | 候选改造 |
| --- | --- | --- |
| **E2E warm p95 > 3s** | OPTW 求解上限超过 3s | 降 `time_limit_seconds`；候选 > 12 时切贪心；缓存解 |
| **Cold 是 warm 的 10×+** | FAISS / bge-small-zh 一次性加载 | 加 warmup 探针；服务启动时预 query 一次 |
| **`search_ugc_evidence` 占大头** | FAISS 检索 + cross-encoder rerank 串行 | 候选缩到 top_k≤32；用更小的 reranker |
| **`solve_constrained_route` 占大头** | Pareto 调 5 次 solve_optw | 把权重 sweep 并行化；缓存同输入解 |
| **`llm` 比 `rule` 慢一个数量级** | 每步都问 LLM | 给决策加 `should_call_llm` 启发式（只在关键岔路调 LLM） |
| **`get_amap_chain` 占大头（去掉 stub 之后）** | 真高德有 N 段路要算 | 段缓存（已有 `amap/cache.py`）+ 并发请求 |

## 9. 与 eval harness 的关系

`bench_latency.py` 和 `eval/run_eval.py` 走的是**同一套场景、同一套 Amap stub、同一个 TestClient**，所以两份报告横向对得上：

- `eval/run_eval.py` ：每场景跑 **1** 次，关心**对不对**（约束 / 忠实度 / 最优解 gap）。
- `bench_latency.py`：每场景跑 **N** 次，关心**快不快**（延迟分位、工具占比）。

两个报告产物分开存（`route_eval.md` 和 `latency_report.md`），避免一份文档既背"质量"又背"性能"。
