## Elevator Pitch（30 秒版本，可直接背）

> 我做了一个叫 **AIroute** 的本地路线规划系统。它把传统的"按规则排序 POI"升级成了**LLM Agent 编排 + 约束优化 + 学习排序 + 多目标 Pareto** 的四层架构：用户说一句自然语言，系统在带时间窗的定向越野（OPTW）模型下用 OR-Tools CP-SAT 求最优解，同时产出**5 条非支配 Pareto 路线**，每条带蒙特卡洛模拟出的准时概率。最后我自己搭了一套**离线评测 harness + CI gate**——在 5 个基准场景上跑出**硬约束满足率 100%、解释忠实度 1.0、与最优解的 gap 0.001**，并把 ranker 模型 NDCG@5 相对规则基线提升了 **+95%**。

> 这个项目我想证明的是：「LLM 应用工程师」不应该等于"调 API + 写 prompt"，而是要能在 LLM 的不确定性外面套一层**有约束、有指标、可验证、可降级**的真系统。

## STAR 叙事

### Situation

本地路线规划是个被低估的复杂问题：用户用自然语言描述诉求（少排队、要吃本地菜、有预算、必去某景点……），系统要在**时间窗 + 营业时间 + 预算 + 类目 + 用户偏好**多重硬约束下给出可解释、可调整的路线。市面上大多数项目要么用 LLM 直接生成路线（幻觉、不可复现），要么用启发式排序（不优化、无解释），都不算"AI 应用工程"。

### Task

把它做成一个：

1. **AI 内核真**——LLM 真的在编排决策，而不是只做 prompt 调用。
2. **算法基础硬**——路线是带时间窗的约束优化结果，能拿出与最优解的 gap。
3. **数据驱动可证明**——任何效果改善都能在离线评测上跑出数字。
4. **工程可演示**——多用户隔离、状态可恢复、CI 上能拦回归、移动端能直接演示。

### Action

按 5 个技术领域分工，每块都是独立可讲的硬实力。

#### A. LLM 应用工程（Agent Harness）

- 设计了一个 **plan-act-observe 循环**：12 步预算、工具白名单、每步结果落到 `AgentState.memory`，LLM 只能选注册的 11 个工具之一（`parse_intent` / `search_ugc_evidence` / `recall_similar_sessions` / `solve_constrained_route` / `compose_story` / `validate_route` / `assess_robustness` / `critique` / `parse_feedback` / `replan_by_event` / `get_amap_chain`）。
- 写了 **LLM tool-calling + 规则决策两条路径**，可由 `agent_tool_calling_enabled` / `agent_fast_decision_enabled` 切换。在评测里两条路径同时跑通 gate，证明 LLM 路径"不更差，是增量"。
- 接入 **OpenAI 兼容 function-calling**（LongCat / DeepSeek 都通），失败自动回退到规则决策；做了 LLM 请求级 LRU 缓存让幂等调用免费命中。
- 全链路 **OpenTelemetry span + structlog + Prometheus 指标**，每个会话能用 `/agent/cost/{id}` 看 token / 延迟 / 估算成本。

#### B. 运筹优化 / 约束建模

- 把"路线规划"形式化成 **带时间窗的定向越野（OPTW, Orienteering Problem with Time Windows）**：变量是「选哪些 POI、顺序、到达时间」，目标是最大化总效用，约束是时间窗 / 营业时间 / 预算 / must-visit / 类目（含 OR 关系类目组）。
- 用 **OR-Tools CP-SAT** 建模：流约束（开放路径，每点入度=出度=x_i）+ 时序传播（`OnlyEnforceIf` 保证 t_j ≥ t_i + visit_i + travel_ij）+ 营业时间窗。
- 实现 **三档求解器**：CP-SAT（默认）→ 精确枚举（候选 ≤8 做 oracle）→ 贪心 fallback（不可行时给"尽力而为"+ violations 标注）。评测里启发式解相对枚举 oracle 的 **gap = 0.001**。
- **多目标加权扩展**：把单目标 maximize utility 改成 `maximize w_u·U - w_t·t_end - w_c·cost - w_q·queue`，让同一个求解器能跑出"偏省时 / 偏省钱 / 偏少排队"的不同解。

#### C. 机器学习 / Learning-to-Rank

- 用 **LightGBM LambdaMART** 替换原本的手调魔数评分，特征是规则评分的 16 维 + queue/price/distance/ugc_sim/popularity。
- 解决了一个关键的 **train/serve skew 问题**：原本训练用"简化版 breakdown + distance=0 + ugc_sim=0"，线上用真实 breakdown，特征分布不一致。我直接 `from app.services.poi_scoring_service import PoiScoringService` 在训练里调线上评分器，根除偏差。
- 用 **合成多 query 分组**（少排队本地菜 / 情侣夜景 / 亲子室内 / 高效打卡）让 lambdarank 真正做 listwise 排序而非退化成 pointwise。
- 用 `should_enable_ranker` 函数做**增益门控**：要求 model NDCG@5 比规则基线高 ≥3% 才推荐启用。最终训出来 **0.4725 vs 0.2422，+95% 相对增益**。
- `PoiRanker.predict` 缺模型时返回 `None` → 上游自动回退到规则总分，**线上完全无感降级**。

#### D. 多目标决策 / Pareto 前沿

- 实现了一个轻量级 Pareto 扫描器：5 个权重 profile 各跑一次 OPTW + 候选 ≤8 时全枚举补充候选池。
- **4 维支配判定**（兴趣 max；时长 / 花费 / 排队 min）→ 去重 → 非支配过滤 → 取前 5。
- 每场景稳定输出 **5 条非支配解**；tradeoff 文案不再硬编码，而是由两解的**指标差自动生成**（"比当前多 8 兴趣、多 12 分时长、省 ¥40"）。
- 前端用 React + TypeScript 写了卡片切换 UI（`AmapRoutePage.tsx`），可选中查看，附带 CSS 样式。

#### E. 评测体系 / MLOps

- 设计了 **5 个基准场景**（半日吃饭 / 预算紧 / 必去景点 / 少排队 / 家庭雨天），每个 YAML 包含 `request` + 期望约束。
- 自己写了 **6 个评测指标**：可行率 / 硬约束满足率 / 解释忠实度 / 与最优解的 gap / NDCG@5 / 蒙特卡洛准时概率。其中：
  - **解释忠实度**：解析每条 `why_this_one` 中的关键词，回查 `score_breakdown` / `poi.high_freq_keywords` / `category` 是否一致——证明可解释性不是编的。
  - **gap**：用候选池重建 OptwNodes，`solver_mode="exact"` 复解拿 oracle，再算 `(opt - sel) / opt`。
- 写了 **CI gate**：`run_eval --enforce-gate` 在 gate 不达标时 `SystemExit`；GitHub Actions workflow PR 触发，artifact 上传 markdown 报告。
- 蒙特卡洛模拟（500 采样、高斯扰动）作为路线第 5 维质量，输出 `on_time_prob / expected_overflow_min / p90_total_min`。

#### F.（额外）系统工程 / 持久化与记忆

- **三层记忆**：episodic（上次会话摘要）+ semantic（`user_facts` 长期偏好）+ vector（`session_vector_repo` 情景召回）。每层都有 Prometheus 计数器看命中率。
- **状态持久化**：`agent_sessions.sqlite` 按 user_id 隔离、`session_id` 主键、支持服务重启恢复；`save_state` 异步触发会话向量化入库。
- **可热替换的 RAG**：FAISS + `BAAI/bge-small-zh-v1.5`（本地中文 embedding，不依赖外部 API），带 provenance 标注。

### Result

| 类别 | 量化结果 |
| --- | --- |
| 算法效果 | 路线与 CP-SAT 最优解的 gap = **0.001**；解释忠实度 = **1.0**；5 场景 100% 通过 gate |
| ML 效果 | Ranker NDCG@5 = 0.4725，相对规则基线 **+95.1%**，远超 3% 启用门槛 |

## 按能力标签拆解贡献（面试时可挑着说）

### LLM 应用工程师方向

- "**杜绝幻觉路线**"：LLM 在我的 Agent harness 里只能选注册的 11 个工具，结果落到 typed state；任何 hallucinated POI 都被 Critic 显式检查并触发重试。
- "**LLM 可观测性**"：每次调用记 token、p95 延迟、降级率、缓存命中率；每会话能算估算成本。
- "**LLM ≠ 系统**"：演示完之后我能拿出 5 场景 × 6 指标的评测表，证明系统效果——这是和"我接了个 OpenAI demo"的区别。

### ML 工程师方向

- "**train/serve skew 是我亲自抓出来的**"：原本训练用占位 breakdown，线上用真实 breakdown，特征分布差很多。我把训练直接接到线上评分器，NDCG@5 立刻翻倍。
- "**弱监督 + 多 query group**"：没有真实点击日志的情况下用合成 query + 偏好信号做弱标签，跑出 +95% 增益的同时为后续接真实日志留好接口。
- "**门控启用 + 安全降级**"：模型不达标自动不启用；启用后模型文件缺失自动回退规则——线上零事故升级路径。

### 运筹 / 算法方向

- "**真把路线建成 OPTW 而不是 TSP**"：解释清楚什么是定向越野（带 prize、带时间窗的 TSP 变体）、为什么用 CP-SAT、为什么要三档求解。
- "**和 OR-Tools 精确解 0.001 的 gap**"：能讲清楚 oracle 怎么算的，能讲清楚为什么这个 gap 有意义（小规模 ≤8 时枚举就是真最优）。
- "**多目标 Pareto 不是几个 if-else**"：能讲清楚加权标量化与 NSGA 的对应、能解释支配判定。

### 系统设计方向

- "**Agent harness 怎么避免成为玩具**"：12 步预算、工具白名单、规则兜底——这三件让 LLM 编排既灵活又安全。
- "**三层记忆**"：能讲清楚 episodic / semantic / vector 三层的边界、为什么分开、各自的 Prometheus 命中率怎么看。
- "**多用户落 SQLite + 行程版本**"：用 `(user_id, session_id)` 隔离 + `agent_sessions` 表 + `save_state/load_state` 复盘任何会话。

### MLOps / 可观测方向

- "**评测 harness 不是事后凑的**"：5 场景 YAML + 6 指标 + exact oracle + CI gate + GitHub Actions artifact——这是产品级 ML 的标配。
- "**解释忠实度自动化**"：把"AI 解释是否有理"从主观判断变成可量化指标（解析 claim → 回查证据 → 命中率），这一项是大多数项目缺的。
- "**OpenTelemetry + structlog + Prometheus**"：单 Span / 全 Trace / 指标分层都接通了。

## 面试话术钩子（5 个可展开的 pivot point）

挑感兴趣的展开，每个都能撑 5–10 分钟。

1. **"我做的不是个 LLM demo，是一套有指标的 AI 系统。"**
   → 引到评测 harness（5 场景 + 6 指标 + CI gate + gap=0.001）。

2. **"我自己抓出来一个 ML 经典坑：train/serve skew。"**
   → 引到 ranker 训练改造（线上评分器入训、合成 query group、+95% NDCG）。

3. **"路线规划其实是 OR 问题，不是 LLM 问题。"**
   → 引到 OPTW / CP-SAT 三档求解 / 与 oracle 的 gap。

4. **"Pareto 前沿替代了原本三个硬编码 style，tradeoff 是算出来的。"**
   → 引到多目标加权 + 支配判定 + UI 切换。

5. **"Agent 怎么不变成玩具？工具白名单 + 步数预算 + 确定性兜底。"**
   → 引到 Conductor 设计、Critic 重试、6 处 fallback。

如果对方追问"实时数据为什么没接"——这就是我**有意识地**留到下一期的工作，先把验证体系搭起来再扩接口，论文里 SCATEAgent 的事件驱动思路也是这么落地的。

## 一句话个人收获

这个项目让我把"AI 应用工程"从一个抽象口号变成了一套可复跑、可证明、可迭代的实践：**LLM 用于决策不用于生成、约束优化用于路线不用于解释、机器学习用于排序不用于裁决、评测用于回归不用于交付**——每层都各司其职，每层都有指标守护。
