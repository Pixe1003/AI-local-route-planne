### 追加：OPT-5（CP-SAT 主求解 timeout 3 → 1.5s）实测不达预期，已回退

试探性把 `conductor.py:_rule_based_decision` 和 `tools.py:_solve_constrained_route` 的 `time_limit_seconds` 从 3 调到 1.5，期望卡掉 `low_queue` 的 p95 长尾。

实测对比（rule, repeats=20）：

| 指标 | 改前 (3s) | 改后 (1.5s) | Δ |
| --- | ---: | ---: | --- |
| warm p95 | 3555 | 4290 | **+21%（更差）** |
| `low_queue` warm p95 | 3772 | 4474 | +19% |
| `solve_constrained_route` warm mean | 1060 | 1144 | +8% |

要进一步压 p95，需要算法级改动而不是参数调优：减少 Pareto profile 数量（5 → 3）、profile `recommend_pool` 内部 531 ms 找子路径、或给 CP-SAT 设 `num_search_workers` 调多核。

## 1. 优化方案（按性价比 / 实施顺序）

### OPT-1 · Pareto 5 次 CP-SAT 并行化  ⭐ 性价比最高

**问题** —— `backend/app/solver/pareto.py:build_pareto_variants` 顺序调 5 次 `solve_optw`（5 个 weight profile），每次 CP-SAT 用 ~250 ms，串行总耗 ~1.3 s。CP-SAT 在 C++ 求解阶段释放 GIL，所以 Python 线程池能真正并行。

**改法** —— `concurrent.futures.ThreadPoolExecutor`，5 worker 并行，结果按 profile name 收回。

```python
# backend/app/solver/pareto.py  改 build_pareto_variants 中段
from concurrent.futures import ThreadPoolExecutor

def build_pareto_variants(
    nodes, travel_minutes, *, solve_kwargs, max_variants=5,
):
    if not nodes:
        return []

    by_id = {node.poi_id: node for node in nodes}
    candidates: list[RouteVariant] = []

    # ===== 并行 5 个 weight profile =====
    def _solve_for(label_and_weights):
        label, weights = label_and_weights
        kwargs = {**solve_kwargs, "weights": weights}
        return label, solve_optw(nodes, travel_minutes, **kwargs)

    with ThreadPoolExecutor(max_workers=len(WEIGHT_PROFILES)) as pool:
        for label, result in pool.map(_solve_for, WEIGHT_PROFILES):
            _append_result(candidates, result, label, by_id, travel_minutes,
                           int(solve_kwargs["start_min"]))
    # ===== /并行结束 =====

    if len(nodes) <= 8:
        candidates.extend(_enumerated_candidates(nodes, travel_minutes, solve_kwargs))
    else:
        candidates.extend(_heuristic_candidates(nodes, travel_minutes, solve_kwargs))

    deduped = _dedupe(candidates)
    frontier = _non_dominated(deduped)
    frontier.sort(key=lambda item: (-item.interest, item.cost, item.queue_min, item.time_min))
    return frontier[:max_variants]
```

**预期收益**
- `solve_constrained_route` warm 均值：**1473 ms → ~400 ms**（5× 并行约打 3.5× 折扣，CP-SAT 有少量启动开销）。
- E2E warm p95：**2598 ms → ~1500 ms**。

**风险与回归**
- CP-SAT 的 `CpSolver()` 对象不是线程安全的；上面代码里每个 `_solve_for` 调用都是 `solve_optw` 自己的局部对象，没问题。
- 验证：`pytest backend/tests/test_pareto.py` + `python -m eval.run_eval --enforce-gate` 不退化（约束满足率仍 1.0、gap 仍 ≈ 0）。
- `pytest backend/tests/test_optw_solver.py` 也跑一次，确认单 solve_optw 行为不变。

**工作量** —— 1 人 · 半天。

### OPT-2 · 启动时 warmup probe  ⭐ cold 体验从 126 s → 3 s

**问题** —— 进程第一次调用要等 126 s（FAISS + bge-small-zh + ranker + sqlite 首次加载）。线上部署后第一个真实用户白等。

**改法** —— FastAPI `lifespan` 启动钩子里跑一次"假调用"，把模型 / 索引 / 缓存全部预热。

```python
# backend/app/main.py  增 lifespan
from contextlib import asynccontextmanager
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动时同步预热，按重要性排序
    try:
        from app.repositories.poi_repo import get_poi_repository
        from app.repositories.ugc_vector_repo import get_ugc_vector_repo
        from app.ml.ranker import get_ranker
        from app.config import get_settings

        settings = get_settings()
        get_poi_repository()                                 # 打开 sqlite
        ugc = get_ugc_vector_repo()                          # 加载 FAISS + bge embedding 模型
        if settings.ranker_enabled:
            get_ranker(settings.ranker_model_path)           # 加载 LightGBM

        # 用一条假 query 让 sentence-transformer 真正 encode 一次（首次 encode 还有 lazy init）
        try:
            ugc.search("warmup", city=settings.default_city, top_k=1)
        except Exception:
            pass
    except Exception as exc:                                 # 永不阻塞启动
        import logging
        logging.getLogger(__name__).warning("warmup failed: %s", exc)
    yield
    # 关闭时无需特殊清理

app = FastAPI(..., lifespan=lifespan)                        # 原 FastAPI(...) 改成这个
```

**预期收益** —— 首请求延迟：**126 s → ≤ 3 s**（cold 代价从用户身上挪到部署阶段）。

**验证**
- 启动后立刻调 `/api/agent/run`，应在 3 s 内返回。
- `bench_latency.py` 再跑一次，§2 Cold start 表里所有 cold 应在 5 s 内。

**工作量** —— 1 人 · 30 分钟。

### OPT-3 · OPTW 解缓存（同请求复发免计算）

**问题** —— 同一个用户在前端切换 Pareto 卡片 / 反复点"重新生成"时，候选集和约束没变，但每次都重解。

**改法** —— `solve_optw` 外面包一层 `functools.lru_cache` 不够（参数有 dict 不可哈希）；写一个手工 cache：

```python
# backend/app/solver/optw_cache.py  新文件
from __future__ import annotations
import hashlib, json
from collections import OrderedDict
from typing import Any

class OptwSolutionCache:
    def __init__(self, max_size: int = 512) -> None:
        self._store: OrderedDict[str, Any] = OrderedDict()
        self._max = max_size

    @staticmethod
    def key(nodes, travel, *, start_min, end_min, budget, must_visit,
            required_categories, required_category_groups, max_stops,
            time_limit_seconds, solver_mode, weights) -> str:
        payload = {
            "nodes": [(n.poi_id, n.utility, n.visit_min, n.price,
                       n.open_min, n.close_min, n.queue_min) for n in nodes],
            "travel": sorted(travel.items()),
            "start": start_min, "end": end_min, "budget": budget,
            "must": sorted(must_visit or []),
            "rc": sorted(required_categories or []),
            "rcg": [sorted(g) for g in (required_category_groups or [])],
            "max_stops": max_stops, "tl": time_limit_seconds,
            "mode": solver_mode, "w": sorted((weights or {}).items()),
        }
        return hashlib.sha1(json.dumps(payload, default=list).encode()).hexdigest()

    def get(self, key: str):
        if key in self._store:
            self._store.move_to_end(key)
            return self._store[key]
        return None

    def put(self, key: str, value) -> None:
        self._store[key] = value
        if len(self._store) > self._max:
            self._store.popitem(last=False)

_GLOBAL = OptwSolutionCache()
```

在 `solver/optw.py` 的 `solve_optw` 入口处查缓存、返回前写缓存。

**预期收益** —— 重复请求 1473 ms → ≤ 5 ms（dict 查找级别）。

**风险** —— 缓存内存膨胀（512 解 × 平均 1KB ≈ 0.5MB，可控）。

**工作量** —— 1 人 · 半天。

### OPT-4 · recommend_pool 内部拆段 + profile  🔍 先看清再改

**问题** —— `recommend_pool` 599 ms 是个黑盒，目前只知道它内部有 FAISS retrieval + rerank + POI 池子构建，但具体哪段最贵不清楚。

# backend/app/services/pool_service.py  内 generate_pool 加 timing（开发期间用）
import time
def generate_pool(self, request):
    t0 = time.perf_counter()
    candidates = self._retrieve(...)
    t1 = time.perf_counter()
    reranked = self._rerank(candidates, ...)
    t2 = time.perf_counter()

### OPT-6 · LLM 决策启发式跳过  ⚠️ 依赖 DAT-2 数据再决定

**问题** —— 如果 LLM 模式真的比 rule 慢很多（比如 +5 s），意味着每步都问 LLM 不划算。

**改法** —— Conductor `_decide` 里加一道启发式：**只在"岔路口"问 LLM**，单链条上的步骤（如 retrieve → score → solve → validate 这种没有选择的）直接走规则。

```python
# backend/app/agent/conductor.py  _decide 内
def _decide(self, state: AgentState) -> Decision:
    fallback = self._rule_based_decision(state)
    settings = get_settings()
    if not settings.agent_tool_calling_enabled:
        return fallback
    if getattr(settings, "agent_fast_decision_enabled", False):
        return fallback
    if not _is_decision_branchpoint(state):     # 新增：非岔路直接走规则
        return fallback
    # 只在岔路时调 LLM
    raw = self.llm.complete_tool_call(...)
    ...

def _is_decision_branchpoint(state) -> bool:
    """判断当前状态是不是真有多个合理下一步。多数线性过渡不算。"""
    # 反馈调整、约束冲突修复、critic 提出多种修复方案 这几种算岔路
    return (
        state.goal.kind == "adjust_route"
        or (state.memory.critique is not None and not state.memory.critique.should_stop)
        or (state.memory.validation is not None and not state.memory.validation.is_valid)
    )
```

**预期收益** —— LLM 模式 E2E 接近 rule 模式（保留 LLM 在关键岔路的"聪明"，但去掉 5–8 次线性过渡的 LLM 调用）。

**工作量** —— 1 人 · 半天。**先做完 DAT-2 拿到数据再决定是否动手**。

### DAT-4 · 并发场景  🔬

**为什么** —— 当前 bench 是单线程顺序调用。线上多用户并发时：
- CP-SAT 多线程会不会互相挤压求解时间？
- FAISS 索引是否线程安全？
- SQLite 读写锁会不会拖慢 agent_sessions 写入？

这些得用并发压测才看得出。

**怎么做** —— 新建 `scripts/bench_concurrency.py`：

```python
# scripts/bench_concurrency.py  骨架
import argparse, time, statistics
from concurrent.futures import ThreadPoolExecutor

def run_one(client, scenario):
    t = time.perf_counter()
    r = client.post("/api/agent/run", json=scenario["request"])
    return (time.perf_counter() - t) * 1000, r.status_code

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--concurrency", type=int, default=10)
    parser.add_argument("--total", type=int, default=50)
    args = parser.parse_args()

    # 用 httpx + uvicorn 真启服务（不能用 TestClient 做并发）
    # 启 uvicorn 子进程，对 http://localhost:PORT/api/agent/run 发请求
    ...

    with ThreadPoolExecutor(max_workers=args.concurrency) as pool:
        results = list(pool.map(run_one, [client]*args.total, [scenario]*args.total))
    latencies = [r[0] for r in results if r[1] == 200]
    print(f"p50={statistics.median(latencies):.0f}ms p95={...}")
```

**期望产物** —— `data/eval/concurrency_report.md`：并发 1 / 5 / 10 / 30 时的 p50/p95，能看出 OPTW + FAISS 在并发下退化曲线。

**工作量** —— 1 人 · 1 天。

### DAT-6 · 加 `--simulate-llm <ms>` flag（可选）  🧪

**为什么** —— 没 LLM key 的人想看到"如果接了 LLM 大概多慢"。

**怎么做** —— bench 脚本里 monkey-patch `LlmClient.complete_tool_call` 做 `time.sleep(ms/1000)` 再返回 fallback。

```python
# scripts/bench_latency.py 增 _simulate_llm
def _simulate_llm(latency_ms: int):
    from app.llm import client as llm_module
    orig = llm_module.LlmClient.complete_tool_call
    def sleeping(self, *a, **kw):
        time.sleep(latency_ms / 1000.0)
        return orig(self, *a, **kw)        # 仍走 fallback，但加了延迟
    llm_module.LlmClient.complete_tool_call = sleeping
```

**期望产物** —— 即便没 key 也能给出"假如 LLM 每步 500 ms"的估算。

**工作量** —— 1 人 · 1 小时。

## 6. 不在本轮做的事（边界说明）

- **CP-SAT 求解器内部参数调优**（如 `num_search_workers`、`linearization_level`）—— 这一项收益不确定且容易踩坑，等 OPT-1/3/5 都做完仍嫌慢再说。

- **CI 上加性能 gate** —— GitHub Actions runner 性能波动大，p95 在那里不可靠，本期不上 gate；要做也得先在自家固定机器上立基线。
